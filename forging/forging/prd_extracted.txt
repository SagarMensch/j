Part 0 — Document Overview & Team Ownership
0.1 Purpose of This Document
This PRD is the single source of truth for building SEQUELSTRING v2.0. Every engineer, designer, and QA analyst must use this document as their primary reference. It defines what to build, how each piece works, and which team member owns each deliverable. Integration conflicts are resolved here.
0.2 Product Vision
SEQUELSTRING is an enterprise-grade AI platform that detects forged, tampered, and duplicate documents before they are approved or paid. It uses six forensic detection engines — ELA, SRM, Noiseprint, DINO ViT, OCR Anomaly, and pHash — backed by a U-Net segmentation model, and surfaced through a Next.js web dashboard. The backend is Python with FastAPI. All relational data is stored in PostgreSQL via Supabase. Neo4j is used where graph-structured data provides analytical advantage. The system targets >= 95% detection accuracy and a verdict latency of under 10 seconds per document.

1.2 Request Lifecycle — Step by Step
Every engineer must understand this flow so they know which system is responsible at each step.
Step 1 — Document submitted. A Finance or HR team member uploads a document through the Next.js portal. The frontend sends a multipart POST directly to Supabase Storage via the Supabase client SDK. The storage bucket path and file metadata are then sent to POST /api/v1/ingest on the FastAPI backend.
Step 2 — FastAPI validates the Supabase JWT from the request Authorization header, extracts the submitter_id and department from the JWT claims, assigns a UUID v4 submission_id, writes an initial ForensicSubmission record to Supabase PostgreSQL with status=QUEUED, and dispatches a Celery task (analyze_document.delay(submission_id)) to the Redis broker. The endpoint returns immediately with { submission_id, status: 'QUEUED' }.
Step 3 — The Celery worker picks up the task. Module 3 (Ingestion) downloads the file from Supabase Storage, extracts text via PaddleOCR, runs PII sanitization, generates the 12-channel forensic tensor (RGB + Noise Residual + ELA + DCT channels), and uploads the tensor to Supabase Storage. Status updated to PROCESSING in Supabase via Realtime broadcast.
Step 4 — Module 1 (Forensic Core) runs all 6 detection engines in parallel using asyncio.gather(). Each engine returns a risk score 0.0–1.0. Scores are weighted and aggregated into the Forensic Risk Score.
Step 5 — Module 2 (U-Net Segmentation) receives the 12-channel tensor and returns a pixel-level segmentation mask identifying tampered regions. The mask PNG is uploaded to Supabase Storage.
Step 6 — Module 4 (Duplicate Detection) runs concurrently with Module 2. It computes MD5, pHash, and text MinHash fingerprints of the document and queries Neo4j for near-duplicate matches across all past submissions.
Step 7 — Module 5 (Report Generator) assembles the complete ForensicSubmission object, writes the final verdict and all scores to Supabase PostgreSQL, generates the PDF report, uploads it to Supabase Storage, and writes the immutable audit log entry. Supabase Realtime broadcasts the status change to COMPLETE, causing the frontend to automatically refresh.
Step 8 — The Next.js frontend receives the Realtime event, fetches the completed report via GET /api/v1/report/{submission_id}, and renders the Analyst Dashboard: risk gauge, per-engine score chart, annotated document with tamper heatmap, OCR anomaly list, and metadata panel. If verdict is CONFIRMED_FORGERY, an alert is dispatched to the submitter's manager.

Part 3 — Backend: Module-by-Module Engineering Specification
Engineering Principles That Apply to All Modules
All backend code is Python 3.11+. Type hints are mandatory throughout. FastAPI's dependency injection system is used for auth, database sessions, and shared services.
All FastAPI endpoints are async. Blocking I/O (file reads, model inference) is dispatched to Celery workers — never run synchronously in an endpoint handler.
No hardcoded secrets. All credentials (Supabase URL, Supabase service role key, Neo4j URI, Redis URL) are loaded from environment variables via pydantic-settings.
Every FastAPI endpoint validates the Supabase JWT using a reusable FastAPI Depends(get_current_user) dependency. Unauthenticated requests receive HTTP 401.
Every service exposes GET /health and GET /metrics. Prometheus scrapes /metrics. If your service doesn't expose these, the DevOps engineer cannot monitor it.
All Supabase PostgreSQL writes that modify multiple rows must use database transactions. Partial writes are a critical bug.
Module 1 — Forensic Engine Core
Owner: Lead AI Engineer
1.1 What This Module Does
Module 1 is the detection brain. It receives the 12-channel tensor path and sanitized text from Module 3, runs all 6 detection engines in parallel, aggregates scores into the Forensic Risk Score, and returns per-engine scores and the verdict. It does not segment tamper regions (Module 2) and does not ingest documents (Module 3).
1.2 The 6 Detection Engines
#
Engine
Implementation Detail
01
ELA
Error Level Analysis — JPEG re-compression map.
The document image is re-saved at quality=90 JPEG. The absolute pixel-wise difference between original and re-saved versions is computed per channel. Edited regions — where someone changed numbers or text using image software — were last compressed at a different quality level, so they produce higher error values than the unmodified surrounding areas. The ELA map is one of the three input channel groups in the 12-channel tensor.
Score: Mean error level in the top-10% highest-error regions, normalized to 0.0–1.0.
02
SRM
Statistical Rich Model — 30-kernel steganalysis.
30 high-pass convolution kernels are applied to the grayscale document image, producing a rich set of residual maps. Each kernel is designed to suppress image content and amplify manipulation artifacts. The resulting 30-channel feature map is passed through a shallow classifier (linear layer) trained on the DocTamper dataset to produce a per-document manipulation score.
Score: Classifier output probability of the TAMPERED class, 0.0–1.0.
03
Noiseprint
Camera fingerprint CNN — noise consistency analysis.
Every scanner or camera imprints a unique, invisible noise pattern on every image it produces. A CNN (DnCNN architecture) extracts the noise residual from the document image by subtracting a denoised version from the original. If all regions of the document came from the same scanner, the residual pattern is spatially consistent. If part of the document was digitally pasted in from another source, its noise pattern differs — Noiseprint measures this inconsistency as an autocorrelation anomaly score.
Score: Spatial inconsistency score from the normalized noise residual correlation map, 0.0–1.0.
04
DINO ViT
Patch attention anomaly — Vision Transformer self-supervised analysis.
The document image is divided into non-overlapping 8x8 patches. facebook/dino-vits8 encodes each patch into a feature vector using its self-supervised Vision Transformer. The cosine similarity between each patch's feature vector and the mean feature vector of all other patches from the same document is computed. Tampered patches — those pasted from another document or heavily edited — produce feature vectors that are dissimilar to the document's natural distribution. The anomaly score is the proportion of patches exceeding a dissimilarity threshold.
Score: Fraction of anomalous patches weighted by their dissimilarity magnitude, 0.0–1.0.
05
OCR Anomaly
Text / amount / date consistency checker.
Operates on the sanitized_text extracted by Module 3 (PaddleOCR). Four checks run in sequence: (1) Amount Consistency — all monetary values are extracted with a regex; line item subtotals are summed and compared to any stated total; a mismatch > 1% tolerance triggers the flag. (2) Duplicate Reference — all reference codes, invoice numbers, and PO numbers are extracted; duplicates within the same document or across same-submitter history trigger the flag. (3) Suspicious Keyword — scan for 'copy', 'test', 'sample', 'draft', 'template', 'void'. (4) Date Plausibility — all dates extracted and validated; dates > 5 years past or in the future are flagged.
Score: Weighted sum of triggered checks. Each check has a configurable weight. Default: amount_mismatch=0.40, duplicate_ref=0.30, suspicious_keyword=0.15, invalid_date=0.15.
06
pHash
Perceptual hash — duplicate submission detector.
A 64-bit perceptual hash (pHash via the imagehash library) is computed from the rasterized document image. The hash is stored in the fingerprints table in Supabase PostgreSQL and simultaneously written to the Neo4j graph as a DocumentNode. On every new submission, a Hamming distance query finds all previously stored pHashes within a configurable distance threshold (default: 8 bits). Distance 0 = exact visual duplicate; distance 1–8 = near-duplicate (minor edits). The pHash score represents how close the nearest match is.
Score: 1.0 - (hamming_distance / 64). Score 1.0 = exact duplicate, 0.875 = 8-bit difference, 0.0 = completely different.
1.3 Score Aggregation & Verdict Logic
The Forensic Risk Score is a weighted average of all 6 engine scores. Default weights: ELA 0.20, SRM 0.20, Noiseprint 0.20, DINO ViT 0.15, OCR Anomaly 0.15, pHash 0.10. Weights are stored in the model_config Supabase table and cached in Redis. They must always sum to 1.0.
Verdict thresholds (configurable via the DevOps UI, stored in model_config): score < 0.40 = CLEAN (auto-approved, certificate issued); 0.40–0.85 = SUSPICIOUS (routed to analyst review queue); score >= 0.85 = CONFIRMED_FORGERY (alert dispatched, submission blocked from approval).

Module 2 — U-Net Segmentation Model
Owner: ML Engineer
2.1 What This Module Does
Module 2 answers: where exactly was the document tampered? It produces a pixel-level segmentation mask showing the precise tampered regions — rendered as the heat map overlay in the Analyst Dashboard.
2.2 Model Architecture & Training
Architecture: segmentation_models_pytorch.Unet with encoder_name='efficientnet-b4', encoder_weights='imagenet', in_channels=12, classes=1.
Input: 12-channel tensor (3 RGB + 3 Noise Residual + 3 ELA + 3 DCT). Resolution: 512x512. Normalized with Albumentations.Normalize().
Output: Single-channel probability mask (512x512). Each pixel value is the probability that the pixel is part of a tampered region.
Post-processing: Binary mask at threshold 0.5. Connected components analysis (scipy.ndimage.label) identifies discrete tampered regions. Each region encoded as a TamperedRegion: { region_id, x, y, width, height, area_px, risk_score }.
Training: DocTamperV1 dataset. DiceLoss. AdamW, lr=2e-4. 24 epochs. Batch size 10. DataParallel for multi-GPU. Early stopping patience 8. Training script: /model/train.py. Model artifact registered in MLflow.
Inference: model loaded from MLflow Model Registry on Celery worker startup. CUDA device if available, CPU fallback with warning.

Module 3 — Document Ingestion & OCR
Owner: ML Engineer
3.1 What This Module Does
Module 3 is the document preprocessor. Every uploaded document passes through this pipeline before forensic analysis begins. It extracts text, generates the 12-channel forensic tensor, sanitizes PII, and detects OCR-level anomalies. It is triggered as the first step in the Celery task chain.
3.2 Ingestion Pipeline — Stage by Stage
Stage 1 — File Validation
File is downloaded from Supabase Storage using the Supabase Python client (supabase.storage.from_('documents').download(storage_path)).
MIME type validation: application/pdf, image/jpeg, image/png, image/tiff, application/vnd.openxmlformats-officedocument.wordprocessingml.document. Any other type fails the job with status=FAILED and an error code of INVALID_MIME_TYPE.
File size check: max 10 MB enforced at this stage as a Celery worker check (size limit also enforced at the FastAPI upload endpoint).
Stage 2 — Text Extraction (OCR)
PDF: PaddleOCR (primary). If the PDF has no embedded text layer (scanned-only), DocTR is used as fallback.
Images (JPEG/PNG/TIFF): PaddleOCR processes the image directly.
DOCX: python-docx extracts paragraphs and table cell text.
Result: raw_extracted_text as a UTF-8 string.
Stage 3 — PII Sanitization
Pass 1 — Regex: Indian PAN, Aadhaar, bank account numbers, IFSC codes, phone numbers, email addresses.
Pass 2 — NER: spaCy en_core_web_trf identifies PERSON, ORG, GPE entities.
Matched entities replaced with typed placeholders: [PAN_REDACTED], [AADHAAR_REDACTED], [BANK_ACCOUNT_REDACTED], [PERSON_NAME_REDACTED].
sanitized_text is written to ForensicSubmission in Supabase. raw_extracted_text is discarded from worker memory immediately.
Stage 4 — OCR Anomaly Pre-Detection
Amount Consistency: All monetary values extracted via regex. Line items summed and compared against any stated total. Mismatch > 1% = ocr_anomaly entry of type AMOUNT_MISMATCH.
Duplicate Reference: All reference/invoice/PO numbers extracted. Duplicates within document or across same-submitter history (queried from Supabase) = DUPLICATE_REF anomaly.
Suspicious Keywords: Scan for 'copy', 'test', 'sample', 'draft', 'template', 'void' = SUSPICIOUS_KEYWORD anomaly.
Date Plausibility: All dates parsed and validated. Dates > 5 years past or future = INVALID_DATE anomaly.
ocr_anomalies JSONB array written to ForensicSubmission. ocr_anomaly_score computed by Module 1 from this array.
Stage 5 — 12-Channel Tensor Generation
Document rasterized to 512x512 PNG at 300 DPI using pdf2image (PDFs) or Pillow (images).
Channel Group 1 (RGB): Standard 3-channel image.
Channel Group 2 (Noise Residual): Per-channel Gaussian blur subtracted from original, result multiplied by 10 and clipped to [0, 255].
Channel Group 3 (ELA): Image re-saved at quality=90 JPEG; absolute difference between original and re-saved, normalized per channel.
Channel Group 4 (DCT): Block DCT on 8x8 tiles. Low (coefficients [0:2, 0:2]), mid ([2:5, 2:5]), and high ([5:, 5:]) frequency bands stored as 3 channels.
All 12 channels stacked and normalized with Albumentations.Normalize(). Saved as a .pt PyTorch tensor file and uploaded to Supabase Storage.
3.3 EXIF Metadata Extraction
EXIF metadata extracted using the exifread library. If no EXIF data is present, exif_stripped is set to True on ForensicSubmission — a significant risk signal used by the OCR Anomaly engine.

Module 4 — Duplicate Detection & Neo4j Graph
Owner: Data Engineer
4.1 What This Module Does
Module 4 detects documents submitted multiple times — even with minor edits to amounts, dates, or reference numbers. It maintains a fingerprint index in Supabase PostgreSQL and a document relationship graph in Neo4j. The Neo4j graph enables fraud pattern queries that relational queries cannot efficiently answer — such as 'find all documents from the same submitter that are visually similar to each other across different vendors'.
4.2 When Neo4j Is Used vs. PostgreSQL

Identify submitters who share duplicate document clusters across departments — multi-hop graph query.

Submitter: Properties: submitter_id (string, unique), department.

4.4 Fingerprinting Strategy
Exact Match (MD5): MD5 hash of raw file bytes. Stored in fingerprints table in Supabase PostgreSQL. O(1) lookup. Detects byte-identical resubmissions.
Near-Duplicate (pHash): 64-bit perceptual hash via imagehash library. Stored in both Supabase PostgreSQL (for range queries) and as a Document node property in Neo4j. Hamming distance < 8 = near-duplicate.
Text Fingerprint (MinHash): 128-band MinHash signature from the sanitized_text token set using datasketch. Jaccard similarity > 0.85 = text duplicate. Stored in Supabase PostgreSQL.
Composite verdict: EXACT_DUPLICATE if MD5 matches. NEAR_DUPLICATE if pHash distance < 8 OR MinHash Jaccard > 0.85. NO_MATCH otherwise.

Module 5 — Report Generation & Audit Trail
Owner: Backend Engineer
5.1 What This Module Does
Module 5 is the final assembly and output step. It generates the human-readable PDF report, writes the immutable audit log entry to Supabase, dispatches CONFIRMED_FORGERY alert notifications, and issues verification certificates for CLEAN documents. It runs as the final step in the Celery task chain after Modules 1, 2, and 4 have completed.
5.2 Report Generation
JSON Report: Serialized ForensicSubmission Pydantic model. Stored in Supabase Storage. Returned by GET /api/v1/report/{id}.
PDF Report: Generated using reportlab. Pages: (1) Cover with verdict badge and risk gauge visual; (2) Per-engine scores table; (3) Annotated document image with tamper region bounding boxes drawn using Pillow; (4) OCR anomaly log; (5) Metadata summary including exif_stripped flag; (6) Analyst notes if a human override has been applied. Stored in Supabase Storage at report_path.
Verification Certificate: CLEAN documents only. One-page PDF stamped with submission_id, MD5 hash, timestamp, document_type, submitter_id, and 'Verified Clean by SEQUELSTRING v2.0'. Stored at certificate_path.
5.3 Audit Log (Supabase PostgreSQL)
An audit_log table in Supabase PostgreSQL records every state change to every ForensicSubmission.
Columns: id (UUID PK), submission_id (FK), event_type (TEXT: SUBMITTED | INGESTION_COMPLETE | ANALYSIS_COMPLETE | VERDICT_ASSIGNED | HUMAN_OVERRIDE | REPORT_ACCESSED | CERTIFICATE_DOWNLOADED | REPORT_EXPORTED), actor_id (UUID — system or human), previous_status, new_status, metadata (JSONB), created_at.
Row-Level Security (RLS) policy in Supabase enforces INSERT only — no UPDATE or DELETE is permitted for any role including service_role. Immutability is enforced at the database layer.
Supabase RLS policy: CREATE POLICY audit_insert_only ON audit_log FOR INSERT WITH CHECK (true); — SELECT is granted to compliance_manager and analyst roles; no UPDATE or DELETE policy defined.
5.4 Alert Notifications (CONFIRMED_FORGERY)
Trigger: verdict = CONFIRMED_FORGERY at the end of the Celery task chain.
Notification channels (configurable per deployment in alert_config Supabase table): email via SendGrid or Resend API, Microsoft Teams incoming webhook, or Slack webhook.
Payload: submitter_id, department, document_type, forensic_risk_score, submission_id, link to /analyst/dashboard?id={submission_id}.
SLA: Alert dispatched within 30 seconds of CONFIRMED_FORGERY verdict assignment.

5.2 CI/CD Pipeline (GitHub Actions)
Branch strategy: main (production), staging (QA), feature/* (development). All PRs require passing tests and a review from the module owner.
Pipeline stages: Lint (ruff + eslint) → Type Check (mypy + tsc) → Unit Tests (pytest + Jest) → Integration Tests → Docker Build → Push to Registry (GitHub Container Registry or AWS ECR) → Deploy to Staging → Smoke Tests → Manual Approval Gate → Deploy to Production.
Next.js deployments to Vercel are triggered automatically on merge to main via Vercel GitHub integration. No Docker needed for the frontend on Vercel.
Supabase migrations are managed with supabase CLI. Migration files are committed to /supabase/migrations/. GitHub Actions runs supabase db push on merge to main using the Supabase service role key from GitHub Secrets.
ML model artifacts are stored in MLflow Model Registry (self-hosted or Databricks). The Celery worker Docker image pulls the 'production' registered model artifact at build time using mlflow.pytorch.load_model().
No secrets in Git. All environment variables are stored in GitHub Actions Secrets and injected at deployment time.

Part 7 — Glossary
Term
Definition
Celery
A Python distributed task queue. Used to run long-running forensic analysis jobs asynchronously so that FastAPI endpoints return immediately. Broker: Redis.
CONFIRMED_FORGERY
The SEQUELSTRING verdict for submissions with forensic_risk_score >= 0.85. Triggers immediate alert dispatch and blocks document from approval.
DINO ViT
Self-DIstillation with NO labels — Vision Transformer. A Facebook AI model that analyses document image patches in a self-supervised manner to detect anomalous regions caused by tampering.
DocTamperV1
The training dataset used for the U-Net segmentation model. Contains real and artificially tampered document images with pixel-level ground truth masks.
ELA (Error Level Analysis)
A forensic technique that re-compresses an image at a known JPEG quality level. Edited regions — where someone changed content using image software — produce higher error values than surrounding unmodified areas.
FastAPI
The Python web framework used for all backend API endpoints. Chosen for its native async support, automatic OpenAPI documentation, and Pydantic integration.
ForensicSubmission
The canonical Pydantic model and Supabase PostgreSQL table that represents a single document submission and all its forensic results. The data contract for all modules.
MLflow
An open-source platform for ML experiment tracking and model versioning. Used to version and serve the U-Net model artifacts.
Neo4j
A graph database used for duplicate document cluster analysis and fraud pattern traversal. Used only where graph structure provides analytical advantage over relational queries.
Next.js
The React framework used for the frontend. Uses App Router with TypeScript. Deployed on Vercel.
Noiseprint
A forensic technique based on the fact that every scanner/camera leaves a unique invisible noise fingerprint on images. Inconsistencies in the noise pattern across a document reveal copy-pasted or digitally edited regions.
OCR Anomaly Engine
The 5th detection engine. Operates on PaddleOCR-extracted text. Checks for amount mismatches, duplicate reference numbers, suspicious keywords, and implausible dates.
PaddleOCR
The primary OCR engine used to extract text from document images. DocTR is used as fallback for image-only PDFs.
pHash (Perceptual Hash)
A 64-bit hash computed from the visual content of a document image. Visually similar documents produce hashes with small Hamming distances. Used for near-duplicate submission detection.
RLS (Row-Level Security)
A PostgreSQL / Supabase feature that enforces access control at the database row level. Used to restrict audit_log to INSERT only, and to filter forensic_submissions to the current user's submissions in the Submitter profile.
SRM (Statistical Rich Model)
A 30-kernel steganalysis feature extraction method. Each kernel is a high-pass filter that suppresses image content and amplifies statistical anomalies caused by digital manipulation.
submission_id
UUID v4 assigned to every document submission at the FastAPI route handler. Primary key throughout all modules and databases.
Supabase
The managed backend platform providing PostgreSQL, Auth (JWT-based SSO), Storage (S3-compatible), and Realtime (WebSocket broadcasts) for SEQUELSTRING.
Supabase Realtime
A WebSocket-based pub/sub system that broadcasts row-level changes from Supabase PostgreSQL to connected Next.js clients. Used for live status updates without frontend polling.
12-Channel Tensor
The normalized input representation of a document for the U-Net segmentation model. Channels: 3 RGB + 3 Noise Residual + 3 ELA + 3 DCT. Dimensions: (12, 512, 512). Generated by Module 3.
U-Net
A neural network architecture for pixel-level image segmentation. Used in SEQUELSTRING to produce per-pixel tamper probability maps. Encoder: EfficientNet-B4. Input: 12-channel tensor.
Verdict
The final classification of a submission: CLEAN (score < 0.40), SUSPICIOUS (0.40–0.85), or CONFIRMED_FORGERY (>= 0.85). Configurable thresholds.
